02. 配置 ​

（1）基础配置 ​

  • 项目的配置文件是位于.vitepress目录下的config.ts，创建项目时会默认生成这个配置文件，现在我们需要改造这个文件

  • 因为这是一篇简单介绍的文章，所以暂时仅需要关注以下几个配置项

    • title：站点的标题
    • description：站点的描述
    • base：站点部署的 base URL，默认为/，后续我们会部署到github的blog仓库，所以这里需要更改为/blog/
    • srcDir：相对于项目根目录的 Markdown 文件所在的文件夹，后续项目会从这个文件夹内读取文件并生成页面
    • srcExclude：需要排除的Markdown文件的目录
    • outDir：项目的构建输出位置
    • cacheDir：缓存文件的目录
    • assetsDir：生成的静态资源的目录
    • themeConfig：主题配置
    tsx
    import { defineConfig } from 'vitepress'

export default defineConfig({
  title: "My Blog",
  description: "My Blog Desc",
  base: '/blog/',
  srcDir: 'blog',
  outDir: '.dist',
  cacheDir: '.cache',
  themeConfig: {...},
})

（2）themeConfig ​

  • themeConfig是用于配置页面的主要配置项，接下来的操作都是围绕这个配置项来完成的

  • 首先，我们暂时只关注这三个选项，主要在于nav和sidebar

    • logo：导航栏上显示的 Logo，配置后会显示在站点标题前
    • nav：导航菜单项的配置
    • sidebar：侧边栏菜单项的配置

  • 配置大致如下

    • text：用于显示文案
    • link：用于匹配文档链接，文档链接与文档的路径一致
    • items：用于配置子级，此时可不写link，子级的每一项也都是text、link，当然子级也可以再有子级
    tsx
    export default {
  themeConfig: {
    nav: [
      { text: 'Guide', link: '/guide' },
      {
        text: 'Dropdown Menu',
        items: [
          { text: 'Item A', link: '/item-1' },
          { text: 'Item B', link: '/item-2' },
          { text: 'Item C', link: '/item-3' },
        ],
      },
    ],
    sidebar: [
      {
        text: 'Guide',
        items: [
          { text: 'Introduction', link: '/introduction' },
          { text: 'Getting Started', link: '/getting-started' },
        ],
      },
    ],
  },
};

（3）自动生成侧边栏、导航栏 ​

  • 在文档较少的情况下，可以一个一个的去写导航栏和侧边栏，但文档数量一旦增多，或者调整了文档的名称，那么维护起来的成本就比较高了

  • 所以，我们需要按照文档所在文件夹递归遍历所有文档，自动生成导航栏和侧边栏

  • 代码如下，可能并不完善，也可到github自行寻找更完善的插件进行生成

    tsx
    // sidebar.ts
import { DefaultTheme } from 'vitepress';
import fs from 'node:fs';
import path from 'node:path';

// 获取文档所在根目录
const blogPath = path.resolve('blog');

// 递归遍历方法
const handleGenerateSidebarWithPath = (pathname) => {
  // 根据传入的路径，获取完整的路径
  const docsPath = path.join(blogPath, pathname);
  // 生成的数据
  const currentSidebarItem: DefaultTheme.SidebarItem = {};
  // 当前路径下所有的文件和文件夹
  const currentFiles = fs.readdirSync(docsPath);

  // 1. 先过滤出所有文件
  const currentFilesOfFile = currentFiles.filter((filename) => {
    if (filename.startsWith('.')) return;

    const filepath = path.join(docsPath, filename);
    const isDir = fs.statSync(filepath).isDirectory();
    return !isDir;
  });
  // 然后根据所有文件生成配置信息
  currentFilesOfFile.forEach((filename) => {
    const filepath = path.join(docsPath, filename);

    const extname = path.extname(filepath);
    const basename = path.basename(filepath, extname);

    if (extname !== '.md') return;

    const dirname = path.basename(path.dirname(filepath));
    !currentSidebarItem?.text && (currentSidebarItem.text = dirname);
    !currentSidebarItem?.items && (currentSidebarItem.items = []);

    if (filename === 'index.md') {
      currentSidebarItem.items.push({
        text: dirname,
        link: `${pathname}`,
      });
    } else {
      currentSidebarItem.items.push({
        text: basename,
        link: `${pathname}/${basename}`,
      });
    }
  });

  // 2. 再过滤出所有文件夹
  const currentFilesOfDir = currentFiles.filter((filename) => {
    if (filename.startsWith('.')) return;

    const filepath = path.join(docsPath, filename);
    const isDir = fs.statSync(filepath).isDirectory();
    return isDir;
  });
  // 然后递归遍历文件夹内的所有文件
  currentFilesOfDir.forEach((filename) => {
    !currentSidebarItem?.text && (currentSidebarItem.text = filename);
    !currentSidebarItem?.items && (currentSidebarItem.items = []);

    const newPathname = `${pathname}/${filename}`;
    const childSidebarItem = handleGenerateSidebarWithPath(newPathname);

    if (JSON.stringify(childSidebarItem) == '{}') return;
    if (childSidebarItem?.items?.length == 0) return;

    const showSubName = filename.replace(/^\d+[.\-_ ]?/, '');
    currentSidebarItem.items.push({
      text: showSubName,
      collapsed: true,
      ...childSidebarItem,
    });
  });

  // 最后返回数据
  return currentSidebarItem;
};

// 将 blog 文件夹下需要显示的文件夹的名称放在这里
// -- 这里我设置的是按照具体路径来生成侧边栏，如果有需要，可自行更改为直接读取整个 blog 文件夹来生成
const currentBlogPath = ['/html', '/css', '/js', '/vue', '/react'];

const sidebarConfig = {};

currentBlogPath.forEach((pathname: any) => {
  sidebarConfig[`${pathname}`] = handleGenerateSidebarWithPath(pathname);
});

// console.log('sidebarConfig ==========>: ', JSON.stringify(sidebarConfig));

export default sidebarConfig;

  • 部署之前，可以先本地运行一下docs:build命令，看是否可以正常打包，然后运行docs:preview，查看是否可以正常运行，如果一切如常可以正常进入到部署的流程了

（2）配置workflow ​

到这里，默认你想将打包构建的结果单独推到另一个仓库部署

  • 第一步：

    • 你需要在你的电脑上新生成一个ssh密钥，需要注意，如果此前你已经生成过了，请另一个路径进行生成，不要覆盖掉以前的
    • 详情请参考文档：生成新的 SSH 密钥

  • 第二步：

    • 将新生成的私钥，添加到当前的VitePress仓库的Actions里面的secrets，具体路径：/settings/secrets/actions
      • 添加secrets的key值，在后面要用到
    • 将新生成的公钥，添加到你要推送的目标仓库的Deploy keys里面，具体路径：/settings/keys
    • 用完之后就可以删除了

  • 第三步：

    • 配置deploy.yml，在当前项目根目录新建.github/workflows/deploy.yml

      yml
      # 构建 VitePress 站点并将其部署到 GitHub Pages 的示例工作流程
name: Deploy VitePress site to Pages

on:
  # 在针对 `main` 分支的推送上运行。如果你使用 `master` 分支作为默认分支，请将其更改为 `master`
  push:
    branches: [main]

  # 允许你从 Actions 选项卡手动运行此工作流程
  workflow_dispatch:

# 设置 GITHUB_TOKEN 的权限，以允许部署到 GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# 只允许同时进行一次部署，跳过正在运行和最新队列之间的运行队列
# 但是，不要取消正在进行的运行，因为我们希望允许这些生产部署完成
concurrency:
  group: pages
  cancel-in-progress: false

jobs:
  # 构建工作
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0 # 如果未启用 lastUpdated，则不需要
      - uses: pnpm/action-setup@v4 # 我当前使用的是 pnpm，如果你是用的是其它的，请自行更换
        name: Install pnpm
        with:
          version: 8
      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      # - name: Setup Pages
      #   uses: actions/configure-pages@v4
      - name: Install dependencies
        run: pnpm install
      - name: Build with VitePress
        run: pnpm docs:build
      # 到这里已经构建好了，现在我们推送到另一个仓库去
      - name: Push to target repo
        uses: s0/git-publish-subdir-action@develop
        env:
          # 换成你自己的目标仓库
          REPO: git@github.com:[yourGithubAccount]/[yourTargetRepo].git
          # 目标分支
          BRANCH: main
          # 要推送的目录，这个目录需要设置成 VitePress 项目设置的输出目录
          FOLDER: .dist
          # 保存在secrets里面生成的ssh私钥
          SSH_PRIVATE_KEY: ${{ secrets.DEPLOY_PRIVATE_KEY }}

04. 结束 ​

  • 到这里应该可以成功部署了，当然每个人遇到的情况可能不尽相同，有的朋友可能会失败，可以多尝试几次

  • 这里的部署同样适用于其它项目，想部署但又不想暴露源码的情况

  • 如果对于Vue这种SPA单页面项目来说，在使用history的路由模式时，在二级路由刷新会导致页面 404

    • 这是由于Github Pages无法处理history模式的前端路由
    • 而当Github Pages发现无法处理的前端路由时，就会跳转至404.html

  • 对于这种情况，我们可以生成一个404.html文件，其内容需要与index.html保持一致

  • 这样即使Github Pages跳转到404页面，也能重新接管前端路由

  • 此时，我们需要做的就是在打包构建完成后，复制一份index.html文件并命名为404.html

    yml
    # 省略...

  - name: Build
  run: pnpm build

  - name: Copy 404
  run: cp dist/index.html dist/404.html
