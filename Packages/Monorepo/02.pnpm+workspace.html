pnpm + workspace ​

01. pnpm ​

  • pnpm是新一代的Node包管理器，相比于yarn和npm，pnpm具有以下优势

    • 安装速度快
    • 基于内容寻址
    • 依赖访问安全性强
    • 磁盘利用率极高
    • 解决了幽灵依赖

    详见另一篇文章pnpm，后续补

（1）拓展 ​

  • pnpm的resolve-peers-from-workspace-root机制
    • 这个机制为配置项，默认为true开启
    • 启用后，将会使用根目录工作区项目的dependencies，去解析子模块中的peerDependencies
    • 确保每个模块中都使用相同版本的peerDependencies

02. workspace ​

  • workspace：pnpm支持Monorepo模式的工作机制叫做workspace

    • 它要求在代码仓库下的根目录存有pnpm-workspace.yaml文件，用于指定哪些目录需要作为独立的工作空间
    • 这个工作空间就是一个子模块，或者说一个子项目，或者说一个npm包
    yaml
    # 这里表示有4个包， package | apps | configs | scripts
packages:
	# 表示将 packages 目录的下一级目录作为包
  - 'packages/*'
  - 'apps/*'
  - 'configs/*'
- 'scripts'

（1）中枢管理 ​

  • 在workspace模式下，代码仓库的根目录通常不会作为一个模块或者说npm包

    • 而是作为一个管理中枢，用于执行一些全局操作、安装一些共有的依赖

  • 初始化

    • 命令：pnpm init

  • 设置用户的全局.npmrc配置

    • 命令：pnpm config set <key> <value>

  • 全局安装依赖

    • 命令：pnpm install
    • 在workspace模式下，子模块内的依赖也会一起安装

  • 安装公共开发依赖

    • 命令：pnpm install -w -D [packageName]
    • -w：表示将模块安装到公共模块中，在monorepo模式下的就是根目录
    • -D：表示安装为开发依赖
      • 等价于--save-dev
    • 安装到devdependencies，表示仅在开发阶段需要使用的node包
    • -S：表示安装为运行依赖
      • 等价于--save
      • 安装到dependencies，表示开发和上线都需要使用到的node包

  • 卸载依赖

    • 命令：pnpm uninstall -w [packageName]

  • 执行脚本

    • 命令：pnpm run dev

  • 限制只允许使用pnpm安装

    • 在package.json中添加scripts
    json
    {
    ...
    "scripts": {
        "preinstall": "npx only-allow pnpm"
    }
}

（2）子模块操作 ​

  • 每一个子模块都需要pnpm init来创建一个package.json文件

    • 需要手动更改每个package.json文件的name字段，每个子模块的name字段不能相同，
    • 这是因为，pnpm并不是通过目录名称来查找，而是通过每个package.json中的name字段来识别不同的模块
    • 通常，推荐使用@仓库名/子模块名来进行命名
    json
    {
  "name": "@will/utils", // 子模块名称，也是引用时使用的名称，需要手动更改
  "version": "1.0.0",
  "description": "",
  "main": "src/index.js", // 根路径指向，需要手动更改
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "type": "module", // node启动模式，需要手动添加
  "keywords": [],
  "author": "",
  "license": "ISC"
}

04. 环境搭建 ​

（1）安装配置 ​

  • 全局安装pnpm

    • 命令：npm i pnpm -g

  • 创建并初始化项目

    • 命令：pnpm init

  • 配置package.json

    • private: true，设置为不需要发布的项目

  • 配置pnpm-workspace.yaml

    • 这个文件会告诉pnpm使用monorepo模式来管理项目
    • 这个文件内容用于指定这个仓库中的各个子模块，每一个子模块都是一个单独的workspace工作空间
    yaml
    packages:
  # 表明 package 目录下的每一个子目录 都是一个独立的模块
  - 'packages/*'
  # 表明 docs 是一个独立的模块
  - 'docs'
  - 'apps/*'
  - 'configs/*'
  - 'scripts'

  • 配置子项目的package.json

    • 每个项目都有自己的package.json文件，用于声明这是一个NPM包目录
    • 需要配置name字段为pnpm-workspace.yaml中配置的项目名称

  • 示例

    json
    // 根目录 package.json
{
    "name": "my-module",
    "private": true,
}

// 子模块1 - /packages/module1
{
    "name": "@my-modul1/module1",
    "version": "0.0.0",
}
